# Система SRS и прогресс мастерства — Kitsune-GENKI

Документация описывает механику интервального повторения (Spaced Repetition System) и систему прогресса для слов и частиц в приложении.

---

## 🎯 Общая концепция

Приложение использует **алгоритм FSRS** (Free Spaced Repetition Scheduler) для оптимального интервального повторения карточек. Каждая карточка (слово или частица) имеет внутреннее состояние, которое определяет, когда её нужно показать снова.

---

## 📊 Структура SRS-карточки

Каждая карточка в `state.srs[cardId]` содержит:

### FSRS параметры (основа алгоритма)

| Параметр         | Описание                                  | Диапазон |
| ---------------- | ----------------------------------------- | -------- |
| `stability`      | Стабильность запоминания (дни)            | 0+       |
| `difficulty`     | Сложность карточки                        | 1-10     |
| `elapsed_days`   | Дней прошло с последнего повтора          | 0+       |
| `scheduled_days` | Запланированный интервал (дни)            | 0+       |
| `reps`           | Количество повторений                     | 0+       |
| `lapses`         | Количество "провалов"                     | 0+       |
| `state`          | Состояние: New/Learning/Review/Relearning | 0-3      |

### Временные метки

| Параметр     | Описание                               | Тип                  |
| ------------ | -------------------------------------- | -------------------- |
| `due`        | Абсолютный timestamp следующего показа | number (ms)          |
| `lastReview` | Timestamp последнего повтора           | number (ms) или null |

### Метаданные поведения карточки

| Параметр        | Описание                                             | Тип     |
| --------------- | ---------------------------------------------------- | ------- |
| `isLeech`       | Флаг "пиявки" (карточка с частыми провалами ≥8)      | boolean |
| `leechNotified` | Пользователь был уведомлён о статусе "пиявки"       | boolean |

> ⚠️ **Важно**: поле `progress` (0-100) больше не используется в новой реализации. Прогресс мастерства определяется через FSRS-параметры: `stability`, `difficulty`, `lapses`.

## 🎲 Качество ответа (Quality Rating)

После каждого ответа пользователя система определяет качество:

| Quality | Название  | Значение | Описание                        |
| ------- | --------- | -------- | ------------------------------- |
| `0`     | **Again** | Снова    | Полный провал, карточка забыта  |
| `3`     | **Hard**  | Трудно   | Вспомнил с трудом, много ошибок |
| `4`     | **Good**  | Хорошо   | Вспомнил правильно, но не сразу |
| `5`     | **Easy**  | Легко    | Вспомнил мгновенно, без усилий  |

### Автоматическое определение quality

Система использует две функции для автоматического расчёта quality на основе ошибок:

#### 1. Базовая логика (для большинства режимов)

```javascript
qualityFromMistakes(mistakeCount, againAt = 2);
```

| Условие                | Quality     | Описание                  |
| ---------------------- | ----------- | ------------------------- |
| 0 ошибок               | `4` (Good)  | Правильный ответ          |
| 1 ошибка (< againAt)   | `3` (Hard)  | Ответ с трудом            |
| ≥2 ошибки (≥ againAt)  | `0` (Again) | Провал, требует повторения |

#### 2. Рисование кандзи (более мягкие критерии)

```javascript
qualityFromDrawingMistakes(mistakeCount);
```

| Условие        | Quality     | Описание                            |
| -------------- | ----------- | ----------------------------------- |
| 0 ошибок       | `5` (Easy)  | Идеальное рисование — редкий бонус  |
| 1-2 ошибки     | `4` (Good)  | Основной успешный результат         |
| 3-5 ошибок     | `3` (Hard)  | Нарисовал с заметными затруднениями |
| ≥6 ошибок      | `0` (Again) | Потребовалось много подсказок       |

### Коррекция quality на основе времени ответа

Система автоматически корректирует качество ответа в зависимости от скорости реакции:

```javascript
adjustQualityByTime(quality, responseTimeMs, mode);
```

**Пороги времени по режимам:**

| Режим                | Быстрый ответ | Медленный ответ |
| -------------------- | ------------- | --------------- |
| Multiple Choice      | ≤3 сек        | ≥10 сек         |
| Particle Quiz        | ≤3 сек        | ≥10 сек         |
| Typing               | ≤5 сек        | ≥15 сек         |
| Sentence Building    | ≤5 сек        | ≥15 сек         |
| Drawing              | ≤10 сек       | ≥30 сек         |

**Правила коррекции:**

- Если `quality = 4` (Good) и время ≤ быстрого → повышается до `5` (Easy)
- Если `quality = 5` (Easy) и время ≥ медленного → понижается до `4` (Good)

**Пример:**

```
Режим: Multiple Choice
Ошибок: 0 → quality = 4 (Good)
Время ответа: 2 секунды (< 3 сек)
Финальный quality: 5 (Easy) ⬆️
```

### Влияние quality на FSRS

Алгоритм FSRS использует quality для расчёта:

- **Нового интервала** (`scheduled_days`) — через сколько дней показать снова
- **Изменения сложности** (`difficulty`) — насколько карточка трудная
- **Стабильности** (`stability`) — насколько надёжно запомнено
- **Счётчиков** (`reps`, `lapses`) — статистика повторений и провалов

## 🎮 Режимы карточек

Приложение использует **5 режимов** показа карточек, выбираемых автоматически на основе содержания и вероятности.

### 1. 🖌️ Режим рисования кандзи (Drawing Mode)

**Назначение**: Практика написания иероглифов от руки с использованием HanziWriter.

#### Условия активации

- Слово содержит хотя бы один символ кандзи (иероглиф CJK: U+4E00–U+9FFF)
- Случайная вероятность: **25%**
- Приоритет: **1-й** (проверяется первым)

#### Механика

1. Пользователь рисует каждый кандзи из слова последовательно
2. Система отслеживает количество ошибок при рисовании каждого штриха
3. После 3+ ошибок на одном кандзи показывается контур для обводки
4. Карточка завершается после успешного рисования всех кандзи в слове

#### Начисление quality

```javascript
qualityFromDrawingMistakes(totalDrawingMistakes);
```

| Условие            | Quality     | Описание                            |
| ------------------ | ----------- | ----------------------------------- |
| 0 ошибок           | `5` (Easy)  | Идеальное рисование — редкий бонус  |
| 1-2 ошибки (всего) | `4` (Good)  | Основной успешный результат         |
| 3-5 ошибок (всего) | `3` (Hard)  | Нарисовал с заметными затруднениями |
| 6+ ошибок (всего)  | `0` (Again) | Потребовалось много подсказок       |

**+ Коррекция по времени:**
- Время ≤10 сек и quality=4 → повышается до 5
- Время ≥30 сек и quality=5 → понижается до 4

> **Примечание**: `totalDrawingMistakes` суммирует ошибки по всем кандзи в слове.

#### Пример

Слово: **学校** (がっこう) — "школа"

- Рисуем 学: 2 ошибки
- Рисуем 校: 1 ошибка
- **Итого**: 3 ошибки → `quality = 3` (Hard)
- Время: 25 секунд (норма) → качество не меняется

---

### 2. ⌨️ Режим ввода с клавиатуры (Typing Mode)

**Назначение**: Практика набора слов на японском с виртуальной клавиатурой (хирагана/катакана).

#### Условия активации

- Слово содержит кандзи
- **И** количество уникальных символов каны ≤ 8 (помещается на виртуальную клавиатуру)
- **И** случайная вероятность: **25%** (после проверки Drawing Mode)
- Приоритет: **2-й**

#### Механика

1. Пользователь видит перевод слова на русском
2. Должен набрать правильное чтение (хирагана/катакана) с помощью 8 кнопок
3. При первой ошибке показывается подсказка: первый символ правильного ответа
4. При второй ошибке показывается полный правильный ответ

#### Начисление quality

```javascript
qualityFromMistakes(typingMistakes, 2);
```

| Попытка                      | Quality     | Описание            |
| ---------------------------- | ----------- | ------------------- |
| 1-я попытка верна            | `4` (Good)  | Набрал без ошибок   |
| 2-я попытка верна            | `3` (Hard)  | Набрал с подсказкой |
| 3-я попытка (автозавершение) | `0` (Again) | Не смог набрать     |

**+ Коррекция по времени:**
- Время ≤5 сек и quality=4 → повышается до 5
- Время ≥15 сек и quality=5 → понижается до 4

#### Пример

Слово: **大学** (だいがく) — "университет"

- Виртуальная клавиатура: `[だ] [い] [が] [く] [た] [き] [ん] [し]`
- Пользователь набирает: `だいかく` ❌ (1 ошибка)
- Подсказка: "Начинается на だ"
- Пользователь набирает: `だいがく` ✅ → `quality = 3` (Hard)
- Время: 8 секунд → качество не меняется

---

### 3. 🔢 Режим множественного выбора (Multiple Choice)

**Назначение**: Выбор правильного перевода из 4 вариантов.

#### Условия активации

- **Либо**: слово не содержит кандзи
- **Либо**: слово не прошло в Drawing/Typing (вероятность **50%**)
- Приоритет: **3-й** (режим по умолчанию)

#### Механика

1. Показывается перевод слова на русском
2. Четыре варианта японских слов (одно правильное + 3 дистрактора)
3. Дистракторы подбираются из слов той же категории (существительные/глаголы/прилагательные)
4. При ошибке кнопка блокируется, пользователь может попробовать ещё раз
5. После 2-х ошибок правильный ответ подсвечивается зелёным цветом

#### Начисление quality

```javascript
qualityFromMistakes(mistakeCount, 2);
```

| Попытка                 | Quality     | Описание               |
| ----------------------- | ----------- | ---------------------- |
| 1-я попытка верна       | `4` (Good)  | Выбрал с первого раза  |
| 2-я попытка верна       | `3` (Hard)  | Выбрал со второго раза |
| 3-я попытка (автопоказ) | `0` (Again) | Не угадал              |

**+ Коррекция по времени:**
- Время ≤3 сек и quality=4 → повышается до 5
- Время ≥10 сек и quality=5 → понижается до 4

#### Пример

Перевод: **"завтрак"**  
Варианты:

- あさごはん ✅
- ひるごはん ❌ (обед)
- ばんごはん ❌ (ужин)
- おやつ ❌ (перекус)

Пользователь выбрал `ひるごはん` → ошибка → выбрал `あさごはん` → `quality = 3` (Hard)  
Время: 7 секунд → качество не меняется

---

### 4. 📝 Режим particle quiz (для частиц)

**Назначение**: Выбор правильной частицы для вставки в предложение.

#### Условия активации

- Карточка является **частицей** (は, の, を, に, で и т.д.)
- Режим всегда активен для частиц

#### Механика

1. Показывается японское предложение с пропуском `[ _ ]`
2. Русская подсказка (перевод предложения)
3. Четыре варианта частиц (одна правильная + 3 дистрактора)
4. Дистракторы — другие разблокированные частицы из уроков

#### Начисление quality

**Аналогично Multiple Choice**:

```javascript
qualityFromMistakes(mistakeCount, 2);
```

| Попытка           | Quality     | Описание                        |
| ----------------- | ----------- | ------------------------------- |
| 1-я попытка верна | `4` (Good)  | Выбрал правильную частицу сразу |
| 2-я попытка верна | `3` (Hard)  | Выбрал со второй попытки        |
| 3-я попытка       | `0` (Again) | Не смог выбрать                 |

**+ Коррекция по времени:** аналогично Multiple Choice (≤3 сек / ≥10 сек)

#### Пример

Предложение: `私 [ _ ] 学生です`  
Подсказка: "Я — студент"  
Варианты: `[は] [の] [が] [を]`  
Правильный ответ: `は` → Тематическая частица

---

### 5. 🧩 Режим Sentence Building (составление предложений)

**Назначение**: Составить правильное предложение из перемешанных слов.

#### Условия активации

- Карточка является частицей
- Альтернатива Particle Quiz (зависит от настроек/вероятности)

#### Механика

1. Показывается русская подсказка (перевод предложения)
2. Пул перемешанных японских слов (включая частицу)
3. Пользователь составляет предложение, нажимая на слова по порядку
4. При ошибке — можно пересобрать предложение заново

#### Начисление quality

**Аналогично Multiple Choice**:

```javascript
qualityFromMistakes(mistakeCount, 2);
```

| Попытка           | Quality     | Описание                   |
| ----------------- | ----------- | -------------------------- |
| 1-я попытка верна | `4` (Good)  | Составил без ошибок        |
| 2-я попытка верна | `3` (Hard)  | Составил со второй попытки |
| 3-я попытка       | `0` (Again) | Не смог составить          |

**+ Коррекция по времени:** ≤5 сек / ≥15 сек

#### Пример

Подсказка: "Я иду в школу"  
Пул слов: `[私] [は] [学校] [に] [行きます]`  
Правильное предложение: `私 は 学校 に 行きます`

## 🐛 Система "пиявок" (Leech Detection)

### Концепция

**"Пиявка"** (leech) — карточка, которую пользователь постоянно забывает, несмотря на многочисленные повторения. Такие карточки требуют дополнительного внимания или изменения подхода к изучению.

### Определение пиявки

```javascript
isLeech(card, threshold = 8);
```

Карточка считается "пиявкой", если:

```
card.lapses >= 8
```

**`lapses`** — количество провалов (quality = 0) за всё время изучения карточки.

### Обработка пиявок

При каждом review система автоматически проверяет статус:

```javascript
handleLeech(card);
```

**Логика:**

1. Если `lapses >= 8` → устанавливается флаг `card.isLeech = true`
2. Если это первое обнаружение → `card.leechNotified = true` (для уведомления пользователя)
3. Планировщик FSRS продолжает работать обычным образом (интервалы не меняются)

### Что делать с пиявками?

**Рекомендации для пользователей:**

- 📝 Проверить качество карточки (неоднозначный перевод, некорректный пример)
- 🔄 Использовать мнемонику или ассоциации
- 📚 Вернуться к теории (урок, грамматика)
- 🎨 Попробовать другой режим изучения

**Будущие улучшения:**

- [ ] Отдельный экран "Проблемные карточки" (фильтр по `isLeech`)
- [ ] Специальный режим повторения пиявок с более мягкими интервалами
- [ ] Предложение добавить карточку в личный словарь с заметками

## 📋 Журнал повторений (Review Log)

### Назначение

Система автоматически сохраняет детальную историю каждого повторения для:

- Анализа прогресса пользователя
- Обучения адаптивных весов FSRS (в будущем)
- Отладки и улучшения алгоритма

### Структура записи

```javascript
{
  cardId: "word_123",              // ID карточки
  quality: 4,                      // Качество ответа (0/3/4/5)
  mode: "multiple-choice",         // Режим карточки
  responseTimeMs: 3420,            // Время ответа в миллисекундах
  timestamp: 1721641234567,        // Timestamp повторения
  previousStability: 5.2,          // Стабильность ДО review
  previousDifficulty: 4.8,         // Сложность ДО review
  previousState: 2                 // Состояние ДО review (Review)
}
```

### Хранение

Журнал сохраняется в **IndexedDB** (`KitsuneGenkiDB` → object store `review_logs`).

### Использование

```javascript
import { setReviewLogger } from './srs.js';
import { saveReviewLog } from './src/review-log.js';

// Подключаем логгер при инициализации приложения
setReviewLogger(saveReviewLog);
```

После этого каждый вызов `SRS.review()` автоматически создаёт запись в журнале.

### Будущие возможности

- 📊 Визуализация прогресса (графики стабильности, точности по режимам)
- 🧠 Обучение персонализированных весов FSRS на истории пользователя
- 🔍 Анализ проблемных категорий слов (существительные vs глаголы)
- ⏱️ Оптимальное время суток для повторений (анализ accuracy по часам)

## ⚙️ Конфигурация FSRS

### Параметры планировщика

Все параметры FSRS вынесены в файл `src/srs-config.js` для удобной настройки:

```javascript
export const SRS_SCHEDULER_CONFIG = Object.freeze({
  requestRetention: 0.9,      // Целевое запоминание 90%
  enableFuzz: true,           // Размытие интервалов
  maximumInterval: 365,       // Максимум 365 дней между повторениями
});
```

### Описание параметров

#### 1. `requestRetention` (целевое запоминание)

**Диапазон:** 0.7 — 0.98  
**Значение по умолчанию:** 0.9 (90%)

Процент карточек, которые пользователь должен вспомнить при повторении.

**Влияние:**

- **Выше (0.95)** → более частые повторения, меньше забытых карточек, больше нагрузка
- **Ниже (0.8)** → реже повторения, выше риск забыть, меньше нагрузка

**Рекомендации:**

- 0.9 — оптимальный баланс для большинства пользователей
- 0.85 — для тех, кто готов допустить больше провалов ради меньшей нагрузки
- 0.95 — для критически важного материала (экзамены, профессиональная лексика)

#### 2. `enableFuzz` (размытие интервалов)

**Значение:** `true` / `false`  
**По умолчанию:** `true`

Добавляет случайное отклонение ±5% к рассчитанному интервалу.

**Зачем нужно:**

- Избегает "кучкования" карточек в один день
- Распределяет нагрузку равномернее по календарю
- Предотвращает эффект "адского понедельника" (все карточки назначены на один день)

**Пример:**

```
Без fuzz: интервал = 10 дней
С fuzz:   интервал = 9.5 — 10.5 дней (случайно)
```

#### 3. `maximumInterval` (максимальный интервал)

**Диапазон:** 30 — 36500 дней  
**Значение по умолчанию:** 365 дней (1 год)

Ограничивает максимальный интервал между повторениями.

**Влияние:**

- Без ограничения FSRS может назначить карточке интервал 5-10 лет
- Для языка это нежелательно — забывается даже хорошо выученное
- Максимум 365 дней гарантирует, что слово повторится хотя бы раз в год

**Рекомендации:**

- 365 — для большинства слов
- 180 (полгода) — для грамматики и частиц (забываются быстрее)
- 90 (3 месяца) — для новичков (поддерживает активное повторение)

### Лимиты нагрузки

```javascript
export const SRS_LOAD_CONFIG = Object.freeze({
  dailyNewCardsLimit: 15,      // Новых карточек в день
  sessionNewCardsLimit: 10,    // Новых карточек за сессию
});
```

#### `dailyNewCardsLimit`

Максимум новых карточек, которые можно **впервые** увидеть за один календарный день.

**Влияние:**

- Защищает от перегрузки (нельзя разблокировать 100 слов и выучить всё за день)
- Поощряет регулярные сессии вместо "запоя"

#### `sessionNewCardsLimit`

Максимум новых карточек в одной сессии.

**Влияние:**

- Ограничивает длительность сессии (~10-15 минут)
- Повторения (review) не ограничиваются — можно повторять сколько угодно

**Пример:**

```
День 1: сессия 1 → 10 новых + 20 повторений
День 1: сессия 2 → 5 новых + 30 повторений (лимит дня: 15)
День 1: сессия 3 → 0 новых + 40 повторений (лимит исчерпан)
День 2: сессия 1 → 10 новых + ... (лимит сбрасывается)
```

## ↩️ Отмена действий (Undo)

### Назначение

Система позволяет откатить последние 10 review операций, если пользователь случайно нажал не ту кнопку или хочет пересмотреть оценку.

### Реализация

```javascript
import { UndoStack } from './src/card-behavior.js';

const undoStack = new UndoStack(10); // Хранит последние 10 действий

// Перед review сохраняем снимок карточки
undoStack.push(card.id, deepClone(card));

// Применяем review
SRS.review(card, quality);

// Если пользователь нажал "Отменить"
undoStack.undo((cardId, previousState) => {
  restoreCardState(state.srs[cardId], previousState);
});
```

### API

#### `new UndoStack(maxSize)`

Создаёт стек с ограничением размера (по умолчанию 10).

#### `push(cardId, previousState)`

Сохраняет снимок состояния карточки перед изменением.

#### `undo(restoreCallback)`

Откатывает последнюю операцию, вызывая callback с `cardId` и `previousState`.

#### `clear()`

Очищает весь стек (например, при завершении сессии).

#### `canUndo` (getter)

Возвращает `true`, если есть действия для отката.

### Важно

- Отмена работает только в рамках текущей сессии
- После закрытия приложения стек очищается
- Отменённые review **не записываются** в журнал повторений

## 🔄 Миграция с SM-2

### Зачем нужна миграция?

Старые версии приложения (до внедрения FSRS) использовали алгоритм **SuperMemo 2** (SM-2) с полями:

```javascript
{
  id: "word_123",
  ef: 2.5,           // Easiness Factor (1.3 — 2.5)
  interval: 7,       // Интервал в днях
  reps: 3,           // Количество повторений
  nextReview: 1721641234567  // Timestamp следующего повторения
}
```

### Функция миграции

```javascript
SRS.migrateSM2ToFSRS(card);
```

**Логика конверсии:**

1. **Easiness Factor → Difficulty**
   ```javascript
   ef [1.3, 2.5] → difficulty [10, 1] (линейно, инвертировано)
   ```
   - Высокий EF (легко) → низкая сложность
   - Низкий EF (трудно) → высокая сложность

2. **Interval → Stability**
   ```javascript
   stability = reps > 0 ? Math.max(1, interval) : 0
   ```
   - Для изученных карточек: стабильность ≈ текущему интервалу
   - Для новых карточек: стабильность = 0

3. **Определение состояния**
   ```javascript
   state = reps === 0 ? State.New : State.Review
   ```

4. **Сохранение абсолютного due**
   - `nextReview` переименовывается в `due`, но timestamp **не меняется**
   - Пользователь не получит внезапный завал просроченных карточек

4. **Сохранение абсолютного due**
   - `nextReview` переименовывается в `due`, но timestamp **не меняется**
   - Пользователь не получит внезапный завал просроченных карточек

### Автоматическая миграция

Миграция происходит **автоматически** при первом запуске после обновления через модуль `src/migration.js`.

**Важно:**
- Функция безопасна: карточки с FSRS-полями не трогаются
- `due` timestamp сохраняется без изменений
- Legacy-поля `ef` и `interval` удаляются после конверсии

## 🔄 Внутрисессионное повторение (SessionManager)

### Концепция "первой попытки"

Система различает:

1. **Первая попытка** (`isFirstAttempt = true`) — применяется полный FSRS алгоритм
2. **Повторные попытки** (`isFirstAttempt = false`) — только внутрисессионное доучивание

### Логика при ошибке

Если пользователь ошибся (`quality < 4`):

1. Карточка **откидывается назад** в очереди на N позиций
2. Показывается снова через несколько карточек
3. FSRS **не применяется** при повторных попытках (штраф уже был при первой)
4. Карточка засчитывается завершённой только при `quality >= 4` (Good/Easy)

### Формула откидывания

```javascript
baseSteps = quality === 0 ? 10 : 15; // Again: -10, Hard: -15
backSteps = Math.max(1, Math.floor(baseSteps / sessionLapses));
```

**Пример**:

- 1-я ошибка (`quality = 0`) → откинуть на 10 позиций
- 2-я ошибка на той же карточке → откинуть на 5 позиций
- 3-я ошибка → откинуть на 3 позиции (динамическое сокращение)

## 📈 Связь с системой опыта (XP)

### Начисление XP

За **каждую завершённую карточку** начисляется:

```javascript
XP_CARD = 1;
```

#### Важно

- XP начисляется **независимо** от quality (0/3/4/5)
- Даже при провале (`quality = 0`) пользователь получает `+1 XP`
- Это мотивирует продолжать обучение даже при ошибках

### Связь XP с уровнями

```javascript
XP_PER_LEVEL = 100; // Опыта на уровень
COINS_PER_LEVEL = 50; // Монет за уровень
```

**Пример прогрессии**:

- 100 карточек → +100 XP → 1 уровень → +50 монет
- 500 карточек → +500 XP → 5 уровней → +250 монет

## 🔧 Техническая реализация

### Файлы

- `srs.js` — Ядро FSRS алгоритма (review, newCard, isDue, миграция SM-2)
- `src/srs-config.js` — Конфигурация FSRS (retention, fuzz, limits)
- `src/srs-helpers.js` — Вспомогательные функции (wordById, isWordUnlocked)
- `src/card-behavior.js` — Leech detection, UndoStack, коррекция quality по времени
- `src/review-log.js` — Журналирование повторений в IndexedDB
- `session-manager.js` — Управление очередью карточек в сессии
- `ui/flashcards.js` — Логика всех 5 режимов карточек
- `src/xp-system.js` — Начисление XP и повышение уровней
- `src/db.js` — IndexedDB обертка для хранения прогресса и SRS-карточек
- `src/migration.js` — Миграция данных из localStorage в IndexedDB (legacy)

### Хранение данных

Все данные пользователя (прогресс, SRS-карточки, настройки) хранятся в **IndexedDB**:

- **База данных**: `KitsuneGenkiDB` (версия 1)
- **Object Stores**:
  - `user_progress` — общие метрики (streak, XP, разблокированные главы/уроки)
  - `srs_cards` — SRS-состояния карточек (stability, difficulty, due_date, reps, lapses)
  - `review_logs` — история повторений для анализа и обучения весов
  - `settings` — пользовательские настройки темы, звука и интерфейса

**Преимущества IndexedDB**:

- ✅ Асинхронные операции (не блокирует UI)
- ✅ Больший лимит хранения (~50MB+ vs 5-10MB у localStorage)
- ✅ Поддержка индексов для быстрого поиска
- ✅ Транзакции для атомарности операций

**Миграция**: При первом запуске после обновления данные автоматически мигрируются из localStorage в IndexedDB (если найдены).

### Основные функции

```javascript
// Создание новой карточки
SRS.newCard(wordId) → { id, stability: 0, difficulty: 0, state: New, due: now }
```

## ❓ Часто задаваемые вопросы (FAQ)

### 1. Параметры инициализации FSRS

**Вопрос**: Используете ли вы стандартные веса FSRS или обучали их на собственном датасете?

**Ответ**: Приложение использует **стандартные веса FSRS** из библиотеки [ts-fsrs](https://github.com/open-spaced-repetition/ts-fsrs) с кастомными параметрами:

```javascript
const scheduler = fsrs(generatorParameters({
  request_retention: 0.9,
  enable_fuzz: true,
  maximum_interval: 365,
}));
```

- **`request_retention: 0.9`** — целевой процент запоминания 90%
- **`enable_fuzz: true`** — размытие интервалов ±5%
- **`maximum_interval: 365`** — максимум год между повторениями
- Все остальные параметры (веса w0–w18) — **стандартные значения FSRS v4.5**
- Веса **не обучались** на собственном датасете

**Почему стандартные веса**:

- FSRS обучен на >20,000 пользователях и >1 млн карточек (Anki dataset)
- Для японского языка базовые веса показывают хорошую точность
- Нет достаточного объёма данных для обучения собственной модели (требуется >10,000 повторений)

**Будущее**: Возможна реализация адаптивных весов на основе статистики конкретного пользователя после накопления достаточного объёма данных (>1000 карточек с историей повторений через `review_logs`).

### 2. Работа с омофонами и синонимами

**Вопрос**: Как режим ввода с клавиатуры (Typing Mode) обрабатывает слова с одинаковым чтением, но разными кандзи? Если пользователь вводит только кану, может ли он случайно ввести чтение другого слова?

**Ответ**: Система учитывает **несколько вариантов чтения** для слов:

#### Механика проверки

```javascript
// Парсинг всех допустимых вариантов чтения
const acceptedAnswers = parseAcceptedAnswers(word.writing);
// Поддерживает: "かう/こう", "かう、こう", "かう、こう"

// Проверка ответа
const isCorrect = acceptedAnswers.some((answer) => cleanKanaString(answer) === userInput);
```

#### Примеры

**Слово**: 買う (покупать)

- Допустимые варианты: `かう`
- Если пользователь введёт `こう` (вариант чтения для 購う/乞う) → ❌ ошибка

**Слово**: 生きる (жить)

- Допустимые варианты: `いきる`
- Если пользователь введёт `せいきる` (другое чтение кандзи 生) → ❌ ошибка

**Слово с несколькими чтениями**: 今日 (сегодня)

- Допустимые варианты: `きょう/こんにち`
- Оба варианта принимаются как правильные ✅

#### Ограничения

⚠️ **Текущая проблема**: Если в словаре не указаны все возможные чтения через `/` или `、`, пользователь может не знать, какой вариант принимается системой.

**Решение**: В файлах уроков (`public/data/lessons/lesson-*.json`) для слов с множественными чтениями указываются все варианты:

```json
{
  "id": "kyou",
  "kanji": "今日",
  "writing": "きょう/こんにち",
  "translation": "сегодня"
}
```

#### Виртуальная клавиатура

Клавиатура генерируется из **всех вариантов чтения**, поэтому если слово имеет 2 варианта, на клавиатуре будут символы для обоих:

```javascript
const keyboardLetters = generateSrsKeyboard(acceptedAnswers);
// Учитывает символы из всех вариантов
```

### 3. Аудио-составляющая и восприятие на слух

**Вопрос**: Запланирована ли интеграция Web Speech API или заранее записанного аудио для карточек? Для языковых SRS восприятие на слух часто выделяют в отдельный приоритетный режим.

**Ответ**: Аудио **уже интегрировано** через Web Speech API, но **не выделено в отдельный режим**.

#### Текущая реализация

**Технология**: [Web Speech API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Speech_API) (встроенный в браузер TTS)

**Файл**: `src/audio-helper.js`

```javascript
// Автоматическое озвучивание при показе карточки
speakJapanese(word.writing, { rate: 0.9 });

// Кнопка повторного озвучивания
<button class="flash-speak" id="flash-speak">🔊</button>;
```

**Где используется**:

- ✅ **Режим рисования** — автоозвучка при показе карточки + кнопка 🔊
- ✅ **Режим множественного выбора** — кнопка 🔊 на лицевой стороне карточки
- ❌ **Режим ввода с клавиатуры** — НЕТ озвучки (по дизайну: пользователь видит русский перевод)
- ❌ **Particle Quiz** — НЕТ озвучки (квиз по грамматике, не словам)

**Параметры**:

- Язык: `ja-JP` (приоритет) или любой `ja-*`
- Скорость: `rate: 0.9` (немного медленнее для лучшего восприятия)
- Системный голос: использует установленный в ОС японский голос

#### Ограничения Web Speech API

**Плюсы**:

- ✅ Работает офлайн (после первой загрузки голоса в ОС)
- ✅ Не требует хранения аудиофайлов (экономия трафика и места)
- ✅ Поддерживается в Chrome, Edge, Safari, Firefox

**Минусы**:

- ❌ Качество зависит от установленных в ОС голосов
- ❌ На Android может быть низкое качество или отсутствие японского голоса
- ❌ Не подходит для точного произношения pitch accent (высоты тона)

#### Почему нет отдельного "Listening Mode"?

**Текущая позиция**: Аудио играет **вспомогательную роль**, а не основную, по следующим причинам:

1. **Фокус приложения** — изучение кандзи, грамматики и письма, не устной речи
2. **Pitch accent** — японский язык критически зависит от высоты тона, TTS не передаёт это точно
3. **Контекст** — слова учатся через письменные примеры (истории, уроки), не аудио-диалоги
4. **Технические ограничения** — Web Speech API не даёт контроль над интонацией

#### Будущие улучшения

##### Вариант 1: Listening Quiz Mode (высокий приоритет)

```

```

**Преимущества**:

- Тренирует восприятие на слух
- Не требует записанного аудио (использует Web Speech API)
- Можно добавить как 6-й режим с вероятностью 15%

**Сложность**: Средняя (добавление в `ui/flashcards.js`)

##### Вариант 2: Предзаписанное аудио (низкий приоритет)

**Источники**:

- [Forvo API](https://forvo.com/) — записи носителей языка
- [JapanesePod101](https://www.japanesepod101.com/) — аудио для N5-N4 слов
- Собственная запись с носителями

**Преимущества**:

- ✅ Точное произношение с pitch accent

**Недостатки**:

- ❌ Требует ~10MB на 1000 слов (не офлайн-френдли)

```
はし [橋] — мост (низкий-высокий)
```

**Реализация**: Интеграция pitch accent данных из словаря JMdict + визуальные метки
