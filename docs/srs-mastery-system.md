# SRS, mastery depth и readiness в Kitsune-GENKI

## Источник истины

Проект использует `ts-fsrs` **5.4.1** (`FSRS-6.0`). Понятия разделены:

- **schedule** — отдельная FSRS-карточка и следующий review;
- **mastery depth** — кумулятивный уровень владения knowledge item;
- **readiness** — готовность вспомнить сейчас, зависящая от retrievability, due и lapse.

Mastery и readiness вычисляются. Изменяемого процента `progress` нет.

## Vocabulary skills и режимы

Один `itemId` имеет отдельные карточки:

| Skill                | Проверка                                        | Режим                |
| -------------------- | ----------------------------------------------- | -------------------- |
| `recognition`        | выбор значения или слова                        | multiple choice      |
| `recall`             | ввод японского слова по значению                | `typing`             |
| `reading-writing`    | воспроизведение кандзи                          | `drawing`            |
| `context-production` | активный ввод слова в контексте или по переводу | `context-production` |

`context-sentence` с вариантами ответа является recognition, а не production.
Particle quiz, sentence building, preview, debug skip и system fallback не могут
изменить vocabulary FSRS.

Recognition сохраняет исторический ID слова. Остальные ID имеют вид
`itemId::skill`. Начатым главам недостающие карточки достраиваются детерминированно.

## Единая typing capability

`src/typing-capability.js` используется одновременно при создании skill-карточки,
выборе режима и рендере ответа.

- катакана нормализуется в хирагану перед сравнением;
- физический ввод хираганой и катаканой равнозначен;
- служебные символы не становятся частью ответа;
- виртуальная клавиатура содержит все символы правильного ответа;
- предел — 12 уникальных символов.

Интеграционный тест проверяет весь каталог из 674 слов. Все слова имеют проходимые
`recall` и `context-production` карточки. Ошибочные `demo` и `totemo` в каталоге
исправлены на `でも` и `とても`.

## Схема FSRS Card

`SRS.serializeCard()` сохраняет все поля `Card` из `ts-fsrs` 5.4.1:
`due`, `stability`, `difficulty`, `elapsed_days`, `scheduled_days`,
`learning_steps`, `reps`, `lapses`, `state`, `lastReview`, а также app metadata.
Timestamps хранятся числами. `SRS.hydrate()` создаёт `Date` только перед вызовом
библиотеки.

Старая запись без `learning_steps` получает детерминированное значение `0`. Legacy
`due` не сдвигается.

## Rating policy

Шкала приложения: `Again=0`, `Hard=3`, `Good=4`, `Easy=5`.

- первая неправильная попытка любого автоматического задания, включая drawing,
  всегда даёт **Again**;
- Again увеличивает FSRS lapse и запускает внутрисессионное доучивание;
- Hard означает успешную первую попытку с явным затруднением и завершает карточку;
- повторная попытка не обновляет FSRS второй раз;
- подсказка или retry не могут дать Easy;
- быстрый multiple choice не повышается автоматически до Easy.

`firstAttemptCorrect` описывает факт первой попытки независимо от Good/Hard.

## Кумулятивная mastery depth

Состояние каждой оси считается отдельно. Новая незрелая ось может блокировать
следующий уровень, но не понижает уже доказанный базовый уровень или score.

Уровни строго иерархичны:

1. **Новое** — нет валидного evidence.
2. **Знакомо** — есть хотя бы один правильный first-attempt review.
3. **Вспоминаю** — выполнено «Знакомо», recall stability ≥ 7 и recall подтверждён
   минимум в два локальных дня.
4. **Уверенно** — выполнено «Вспоминаю», recognition и recall stability ≥ 30,
   recall accuracy последних 20 событий ≥ 80%.
5. **Освоено** — выполнено «Уверенно», recall stability ≥ 90 и активный production
   stability ≥ 90, accuracy ≥ 80%, успехи минимум в два локальных дня.

Для item без применимого context-production production axis выбирается из
`reading-writing`, затем `recall`. Поэтому отсутствие неприменимой оси не делает
высший уровень принципиально недостижимым.

Из evidence исключаются undone, preview, debug, fallback, system events и режимы,
не соответствующие skill. Старый `context-sentence` multiple choice не подтверждает
production.

Score привязан к достигнутому уровню: 0, 15–39, 40–69, 70–99, 100. Прогресс к
следующему уровню считается только по его обязательным осям.

## Readiness

Retrievability не входит в условия mastery depth. Поэтому прошедшее время не
понижает `Освоено` до `Уверенно`.

Отдельный статус имеет значения:

- `Ещё не проверено`;
- `Готово`;
- `Пора освежить` — карточка due, есть недавний lapse или R ниже порога уровня;
- `Требуется подтверждение` — оценочное legacy knowledge.

UI показывает уровень и readiness отдельно.

## Legacy migration

Версия state — **5**.

- v3→v4 сохраняет полную FSRS-схему и помечает старый `progress` как
  `legacyMasteryEstimated`;
- v4→v5 создаёт `masteryArchive` и сворачивает старый review journal;
- старая stability не выдаётся за чистое новое evidence, потому что могла включать
  смешанные режимы;
- изученная legacy-карточка отображается как `Ранее изучалось · Требуется подтверждение`,
  а не `Новое · 0`;
- новые чистые skill review постепенно заменяют оценочный статус.

Формат полного бэкапа — **5.0**; импорт 4.0, 3.0 и 2.0 поддерживается.

## Bounded review journal и Undo

`state.reviewEvents` ограничен **20 событиями на item**. Полные `previousCard` и
`nextCard` остаются только у последних **10** активных событий — это предел Undo.
Более старые события сворачиваются в `state.masteryArchive`: сохраняются skill
evidence, два успешных локальных дня, счётчики и последний lapse.

Таким образом основной `app_state` имеет ограниченный размер относительно каталога,
а каждое сохранение не переносит бесконечную историю полных Card snapshots.

Undo восстанавливает полный `previousCard`, помечает соответствующий `eventId`
через `undoneAt` и сохраняет карточку с журналом одним snapshot. Последние десять
review можно отменять и после reload.

Legacy IndexedDB store `review_log` остаётся только для совместимости старых
бэкапов и диагностических вызовов.

## Локальные дни и лимиты

Все дневные ключи создаются из локальных компонентов `Date`, без UTC slicing.
Dashboard и запуск сессии используют один `countAvailableCardsForSession()`.
Дневной лимит ограничивает новые карточки, но не блокирует уже введённые review.
