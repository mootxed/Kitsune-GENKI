# FSRS Review Log

## Что хранится

Каждый успешный вызов `SRS.review()` добавляет append-only запись в IndexedDB store
`review_log`:

```js
{
  id,                     // autoIncrement, задаёт порядок при равных timestamp
  cardId,
  quality,                // шкала приложения: 0 / 3 / 4 / 5
  mode,
  responseTimeMs,
  timestamp,
  previousStability,
  previousDifficulty,
  previousState,
}
```

`previous*` копируются до вызова планировщика и до мутации карточки. Внутрисессионные
попытки доучивания не попадают в журнал, потому что текущая логика не вызывает для них
FSRS review. Системные пропуски и debug-пути помечаются режимами `system-fallback` и
`debug-skip`; их следует исключать из обучающей выборки.

Журнал включён в полный JSON-бэкап как `data.reviewLog`. Программный доступ:

```js
import { getReviewLogs } from '../src/review-log.js';

const rows = await getReviewLogs();
```

## Подготовка данных для оптимизатора

В проекте установлен `ts-fsrs@5.4.1` (`FSRS-6.0`, 21 вес). Сам `ts-fsrs` выполняет
планирование; для будущего Node-пайплайна обучения предназначен
`@open-spaced-repetition/binding` и его `computeParameters()`.

1. Удалить `system-fallback`, `debug-skip`, повреждённые и дублирующиеся события.
2. Сгруппировать записи по `cardId`.
3. Внутри карточки сортировать по `timestamp`, затем по `id`.
4. Преобразовать quality в Rating без изменения runtime mapping:
   `0 → 1`, `3 → 2`, `4 → 3`, `5 → 4`.
5. Рассчитать `delta_t` в днях между соседними review тем же способом, который
   использует закреплённая версия FSRS; для первой записи новой карточки — `0`.
6. Сохранить same-day события (`delta_t = 0`) при включённом short-term обучении.

Оптимизатор получает массив историй. По официальному примеру для каждого review
добавляется накопительный префикс истории, а не только одна финальная строка на карточку:

```js
[
  [{ rating: 3, delta_t: 0 }],
  [
    { rating: 3, delta_t: 0 },
    { rating: 1, delta_t: 4 },
  ],
];
```

Для текущего short-term поведения параметры оптимизатора должны включать
`enableShortTerm: true` и `numRelearningSteps: 1`. Версию optimizer следует закрепить
в lockfile вместе с версией планировщика.

Для стандартного обучения весов используются последовательности `rating + delta_t`.
`mode` и `responseTimeMs` полезны для анализа срезов и качества данных, но не являются
входами базового оптимизатора. `previousStability` и `previousDifficulty` нужны для
аудита: их нельзя использовать как целевые значения, поскольку они рассчитаны старыми
весами.

Если первая доступная запись карточки имеет `previousState !== New` или
`previousStability > 0`, история обрезана слева. Такую карточку лучше исключить из
обучения с нуля. Обучение и проверку следует делить хронологически, сравнивать новые
веса с текущими на holdout-наборе и применять только валидный набор весов для той же
версии FSRS.
